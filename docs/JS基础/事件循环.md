## 题
### 以下代码，click事件分别是浏览器触发和脚本触发的时候，结果有什么不同？

```js
const nextClick = new Promise(resolve => {
    link.addEventListener('click', resolve, { once: true})
});



nextClick.then(event => {
    event.preventDefault();
});

link.click();
```

1. 物理点击（用户触发）
当用户点击链接时，浏览器会将事件放入任务队列（Task Queue）。

事件触发：执行栈（Stack）为空。

监听器执行：浏览器从队列取出任务，执行 addEventListener 里的回调。

微任务入队：回调中调用了 resolve()，对应的 .then()（微任务）进入 Microtask Queue。

监听器结束：回调执行完毕，此时 JS Stack 清空。

微任务检查点（Check Point）：根据规范，只要 Stack 一空，就要清空所有微任务。于是 .then() 里的 event.preventDefault() 执行。

默认动作：微任务执行完后，浏览器检查 defaultPrevented 标志，发现被阻止了，于是放弃跳转。

2. 脚本点击（link.click()）
当你调用 link.click() 时，情况发生了剧变，因为这是一个同步方法。

脚本运行：此时你的主脚本（即包含 link.click() 的代码）正在 JS Stack 中。

同步触发：.click() 会同步触发监听器。

监听器执行：执行回调，resolve() 被调用，微任务进入队列。

监听器返回：回调执行完，但此时 .click() 还没结束，你的主脚本仍在 Stack 中。

浏览器默认动作：因为 .click() 是同步的，浏览器在 .click() 方法返回之前，就会尝试执行“跳转”这个默认动作。

微任务滞后：由于此时 JS Stack 不为空（主脚本还在跑），微任务队列不会被检查。等到主脚本运行完、Stack 清空时，浏览器早已完成跳转动作，preventDefault() 已经“迟到”了。
